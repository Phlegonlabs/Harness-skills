# Multi-Surface Workflow

## Overview

Projects managed by the harness orchestrator can span multiple product surfaces (web, mobile, API, CLI, desktop, agent). Each surface lives in its own workspace under `apps/` and shares a common design system, but milestones define surface-specific UI specs independently. The orchestrator routes agent dispatch based on surface type and task classification so that the right specialist handles each unit of work.

## Surface Registration

Surfaces are registered in `projectInfo.types` and normalized by `normalizeProjectTypes()` in `runtime/surfaces.ts`. The function always prepends `monorepo` and deduplicates the remaining entries. The `SURFACE_WORKSPACE_MAP` constant maps each `AddableSurface` to its default workspace name:

| Surface Type | Workspace Directory |
|-------------|-------------------|
| `web-app` | `apps/web` |
| `ios-app` | `apps/ios` |
| `android-app` | `apps/android` |
| `api` | `apps/api` |
| `mobile-cross-platform` | `apps/mobile` |
| `cli` | `apps/cli` |
| `agent` | `apps/agent` |
| `desktop` | `apps/desktop` |

`surfaceWorkspaceList()` discovers actual workspace directories by reading `apps/` from the filesystem. If no directories exist yet, it falls back to the workspace map for the declared project types. The `ADDABLE_SURFACES` array lists all valid surface types that can be added after initial project creation.

## Design System Sharing vs Per-Milestone Independence

A single global design system lives at `docs/design/DESIGN_SYSTEM.md` (the `DESIGN_SYSTEM_PATH` constant in `material-policy.ts`). It defines tokens, typography, color scales, and component conventions shared across all surfaces.

Each milestone with UI tasks gets its own spec at `docs/design/{milestoneId}-ui-spec.md`, resolved by `currentMilestoneSpecPath()`. This function lowercases the current milestone ID and checks for the file on disk via `existingRef()`. The milestone spec captures screen-level layouts, interaction flows, and responsive breakpoints for the active surface only.

The constraint enforced by `getAgentMaterialPolicy()` for the `frontend-designer` agent is explicit: "Define or update the design system only for the active product surface." This prevents cross-surface bleed during a single milestone.

## Agent Dispatch Routing

The dispatcher in `runtime/orchestrator/dispatcher.ts` routes agents based on task properties and surface type:

| Condition | Dispatched Agent | Routing Logic |
|-----------|-----------------|---------------|
| UI milestone missing design system or spec | `frontend-designer` | `needsFrontendDesigner()` checks `isUiProject()` then `needsDesignSystem()` or `needsMilestoneSpec()` |
| Current task has `isUI: true` | `execution-engine` | Post-action triggers `design-reviewer` for approval |
| Current task has `isUI: false` | `execution-engine` | Post-action triggers `code-reviewer` instead |
| Design review requested | `design-reviewer` | `dispatchDesignReview()` validates task is UI and IN_PROGRESS |
| Code review requested | `code-reviewer` | `dispatchCodeReview()` validates task is non-UI and IN_PROGRESS |

## Material Policy per Surface

`packetRefsFor()` in `material-policy.ts` assembles required and optional file references for each agent. Surface awareness shows up in several places:

- **frontend-designer**: Required refs include `prdRef("design")` and `architectureRef("frontend")`. Optional refs include `DESIGN_SYSTEM_PATH` and `currentMilestoneSpecPath()`.
- **execution-engine**: Design materials (`DESIGN_SYSTEM_PATH`, milestone spec) are attached only when `task.isUI` is true.
- **design-reviewer**: Always requires `DESIGN_SYSTEM_PATH` and the milestone spec. Warns if either is missing.
- **code-reviewer**: Never receives design materials. Its required refs focus on `prdRef("requirements")` and `architectureRef("rules")`.

## UI Project Detection

`isUiProject()` from `runtime/shared.ts` returns true when `projectInfo.types` includes any of the values in `UI_PROJECT_TYPES`: `web-app`, `ios-app`, `android-app`, `mobile-cross-platform`, or `desktop`. Non-UI surfaces like `api`, `cli`, and `agent` skip the entire design loop.

The `hasAgentSurface()` helper returns true when `types` includes `agent`. Projects with an agent surface get additional scaffolding: a `SKILLS.md` catalog and `skills/api-wrapper/SKILL.md` contract, generated by `getManagedSkillSpecs()` in `runtime/generated-files.ts`.

## Cross-Surface Coordination

When a project declares multiple surfaces (e.g., `["web-app", "api"]`), each milestone's tasks specify their target surface through the `isUI` flag. The orchestrator does not dispatch `frontend-designer` or `design-reviewer` for backend-only milestones, even if the project as a whole is a UI project. This keeps execution focused on the surface that the current milestone actually touches.

## Surface-Aware Scaffold Files

The scaffold generator uses surface types to decide which files to instantiate. Skip logic in `generated-files.ts` excludes surface-specific artifacts that do not match the declared types:

- Swift and Xcode files are skipped when `types` does not include `ios-app`
- Docker files are skipped when `types` does not include `api` or `web-app`
- `docs/design/DESIGN_SYSTEM.md` is skipped when `isUiProject(types)` returns false

Shared infrastructure lives in `packages/shared/` and is available to all surfaces. The `projectTypeSummary()` function joins surface labels (e.g., "Monorepo + Web App + API") for display in generated README and public docs.
